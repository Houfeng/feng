# Feng 语言模块规范

本文档用于补充 [feng-language.md](./feng-language.md) 中的模块系统概要说明,聚焦 feng 语言的模块声明、模块导入与模块使用规则。

## 1 模块系统概览

- Feng 使用 `mod` 进行文件级模块声明,使用 `use` 导入外部模块或二进制包。
- 模块系统支持多级命名空间,通过 `.` 进行层级分隔。
- 模块声明与模块导入在文件内有固定位置约束。
- 模块具备可见性控制,默认包外不可见,需显式声明公开访问权限后才可被外部导入。
- 模块既可用于源码组织,也可用于 feng 自有二进制包和 C ABI 兼容包的导入。

## 2 模块声明规则

- `mod` 必须是文件中的第一个非空且非注释行,无花括号、无代码块。
- 一个文件只能属于一个 `mod`; 同一个 `mod` 可以分布在多个文件中,编译后自动合并。
- 模块名支持 `.` 分隔多级命名空间。
- 同一个 `mod` 在单文件或多文件合并后,模块级 `type` 名称必须唯一。
- 同一个 `mod` 在单文件或多文件合并后,模块级顶层 `fn` 可形成重载集合,但重载只在该模块自身声明的同名函数之间成立。
- 若同一模块内两个同名顶层 `fn` 的参数类型在鸭型规则下存在可重叠匹配,则该重载集合非法,编译期报错。

重名约束说明:

- 若同一模块在多个文件中分别声明同名 `type`,编译期报错。
- 若同一模块在多个文件中分别声明同名顶层 `fn`,只有在参数类型集合互不重叠时才可构成合法重载; 若仅返回值不同、参数列表完全相同,或参数类型在鸭型规则下存在可重叠匹配,则编译期报错。
- 上述约束在单文件内同样成立,不能通过拆分文件规避。

## 3 模块可见性

- 模块可见性遵循 `pr` / `pu` 规则。
- `mod` 默认等价于 `pr mod`,包外不可见。
- 需要对外暴露时,必须显式使用 `pu mod` 声明公开模块,外部代码才可导入。

可见性示例:

默认私有模块:

```feng
mod app.internal.cache;
```

显式私有模块:

```feng
pr mod app.internal.cache;
```

公开模块:

```feng
pu mod app.api.user;
```

导入公开模块:

```feng
use app.api.user;
```

## 4 模块导入规则

- `use` 必须位于模块声明之后、类型与函数定义之前。
- `use` 会把目标模块或包中的公开 `type` 与公开模块级顶层 `fn` 引入当前模块的可见作用域。
- 支持导入源码模块、feng 自有二进制包、C ABI 兼容包。
- 支持使用 `use 模块名 as 别名` 为导入目标声明文件内别名。
- 通过 `use` 引入到当前模块作用域的同名 `type` 或同名顶层 `fn`,不会与其他来源的声明共同组成新的重载集合; 只要在当前可见作用域内形成重名,即编译期报错。
- 当前模块内新声明的 `type` 或顶层 `fn`,若与已通过 `use` 引入的可见名称重名,同样编译期报错。

别名导入规则:

- `as` 别名仅在当前文件内生效,不改变被导入模块的真实模块名。
- 使用别名后,该导入目标的公开成员通过 `别名.成员名` 访问,不再以短名直接注入当前作用域。
- 同一文件内的 `use` 别名必须唯一,不能与其他 `use` 别名重名。
- 若未使用 `as`,则目标模块的公开成员以短名直接进入当前模块作用域。

```feng
use my.app.user as user;
use my.utils.math as math;

let item = user.get_current();
let total = math.add(1, 2);
```

## 5 模块级初始化顺序

模块级 `let` / `var` 变量若带有初始化表达式,则属于模块级初始化逻辑的一部分。模块级初始化在程序启动时自动执行,每个模块只初始化一次。模块级变量采用两阶段初始化: 先为所有模块级变量建立存储并写入该类型默认值,再按初始化顺序执行各自的初始化表达式。

规则说明:

- 单个文件内的模块级变量按自上而下顺序初始化。
- 同一 `mod` 分布在多个文件中时,语言规范不对这些文件中的模块级 `let` / `var` 初始化先后顺序作出保证。
- 编译器内部可以自行选择稳定实现策略,例如按文件的标准化相对路径字典序处理,但该顺序不属于对外承诺的语言语义,用户代码不得依赖。
- 一个 `mod` 若 `use` 了其他 `mod`,则总是先初始化这些 `use` 目标,再初始化当前 `mod` 自身。
- 同一文件内存在多个 `use` 声明时,按声明书写顺序依次初始化对应模块。
- 一个模块即使在多个位置被 `use`,也只在首次需要时初始化一次; 后续 `use` 或访问该模块成员都不会重复触发初始化。
- 在初始化过程中,若读取到某个尚未完成初始化表达式的模块级变量,则读取结果为该变量当前存储中的值; 在其初始化表达式执行完成前,该值初始情况下为类型默认值。
- `use` 循环不会单独导致编译期报错; 若循环中的某一侧在对方完成初始化前读取其模块级变量,则读取到的也是该变量当时的当前值,初始情况下为类型默认值。

重点提示:

- 同一 `mod` 中,若模块级 `let` / `var` 分布在不同文件,则这些初始化的相对顺序不受语言规范保障。
- 一个 `mod` 若 `use` 了其他 `mod`,则总是先初始化这些 `use` 目标,再初始化当前 `mod` 自身; 多个 `use` 按声明顺序初始化。
- 一个模块即使在多个位置被 `use`,也只会执行一次模块级初始化,不会重复初始化。
- 同一 `mod` 跨文件依赖与 `use` 循环采用同一套语义: 若读取尚未完成初始化的模块级变量,则得到该变量当前值,初始情况下为类型默认值。

```feng
pu mod app.main;

use app.config;
use app.runtime;

let name = app.config.load_name();
let boot = app.runtime.start(name);
```

在上例中,初始化顺序为:

1. 先初始化 `app.config`
2. 再初始化 `app.runtime`
3. 最后按当前文件自上而下初始化 `name` 与 `boot`

若 `app.config` 或 `app.runtime` 自身分布在多个文件中,则各自模块内部跨文件的模块级初始化相对顺序不属于语言保证; 在该过程中若提前读取到尚未完成初始化的模块级变量,得到的是该变量当前值,初始情况下为类型默认值。

## 6 书写顺序与格式建议

- `mod` 声明后进入 `use` 导入段时,建议在两者之间保留一个空行。
- 多个 `use` 声明之间不空行,保持连续排列。
- `use` 导入段与下方类型、函数或其他实现代码之间建议保留一个空行。

书写示例:

```feng
pu mod my.app.user;

use my.utils;
use my.core;
use my.app.service as service;

type User {
    pu var name: string;
}

fn load() {
    service.run();
}
```
