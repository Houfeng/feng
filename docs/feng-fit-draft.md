新增 `spec` 与 `fit` 关键字后，两者不应分散说明。`spec` 负责声明契约形状；`fit` 既负责声明具体类型对契约的显式适配，也保留 `fit A` 这一类面向类型自身的能力注入形式。

# 语言设计规约：`spec` 与 `fit`

## 1 设计目标

- 默认不开放。仅因两个类型结构相似，不自动建立可赋值、可传参或可返回的关系。
- 显式声明。类型能力的建立必须通过 `fit` 明确写出。
- 非侵入适配。允许在不修改原始 `type` 定义的前提下，使其满足某个 `spec`。
- 文档成对说明。`spec` 给出契约目标，`fit` 给出显式适配关系，因此两者应放在同一份说明文件中。

## 2 `spec` 的职责

- `spec` 用于描述形状或契约，不提供实现体。
- `spec` 可以描述对象形状，也可以描述可调用形状。
- `spec` 可作为参数类型、返回类型、成员类型和其他类型位置中的引用目标。

示例:

```feng
spec Named {
    let name: string;
    display(): string;
}

spec Click(): void;
```

补充规则:

- `spec` 只声明成员与签名，不声明构造函数实现、方法实现或对象布局细节。
- `spec` 中无函数体的行为签名不使用 `fn` 关键字; `fn` 仍只用于定义带函数体的实体函数、成员方法和构造函数。
- `spec` 本身不等于 `type`。`type` 负责给出具体对象定义，`spec` 负责给出契约目标。

## 3 `fit` 的职责

- `fit` 有两类合法用法: 契约适配与自扩展。
- 契约适配使用 `fit A for B`; 其中左侧 `A` 必须是具体类型，右侧 `B` 必须是 `spec`。
- 自扩展使用 `fit A`，用于为类型 `A` 自身补充额外能力。
- 不再支持“任意类型对任意类型”的开放式 `fit`。

示例:

```feng
fit User for Named;

fit User {
    fn say_hi(self) {
        print("Hello, " + self.name);
    }
}
```

当具体类型尚未直接满足目标 `spec` 时，可在 `fit` 声明中显式补齐目标契约要求的行为:

```feng
fit ThirdPartyUser for Named {
    fn display(self): string {
        return self.first + " " + self.last;
    }
}
```

补充规则:

- `fit` 只建立“类型 A 满足契约 B”的关系，不改变 `A` 原本的定义类别。
- 在 `fit A for B` 的契约适配形式中，`fit` 块只能补齐目标 `spec` 已声明的行为，不应用来向 `A` 注入与目标 `spec` 无关的新能力。
- 在 `fit A` 的自扩展形式中，`fit` 块用于向 `A` 补充额外能力，不要求先存在目标 `spec`。

## 4 `spec` 与 `fit` 的配合关系

- `spec` 先定义契约边界，`fit` 再声明哪些具体类型进入该契约。
- 若没有可见的 `fit` 声明，即使某个 `type` 在结构上已经“看起来满足”某个 `spec`，编译器也不自动建立该关系。
- 只要当前作用域同时可见具体类型 `A`、目标 `spec B` 以及对应的 `fit A for B` 声明，`A` 就可以在要求 `B` 的位置上使用。
- `fit A` 不建立新的契约关系，而是直接把新增能力附着在 `A` 上。

这意味着 `fit` 主要围绕 `spec` 承担契约适配职责，同时保留面向类型自身的自扩展特例。因此把 `spec` 与 `fit` 放在同一份说明文件中仍然更清晰，也更符合当前设计收敛方向。

## 5 作用域与导出规则

- `fit` 的可见性规则与普通声明一致。
- 跨模块使用时，必须显式 `use` 包含该 `fit` 声明的模块。
- 若一个 `fit` 声明需要导出到包外，当前包必须拥有左侧具体类型 `A` 或右侧目标 `spec B` 之一。
- 不满足导出条件的 `fit` 声明只在当前包内生效，不应把关系扩散到包外。

## 6 明确排除的旧设计

以下旧方向不再采用:

- 仅凭结构自动成立的开放式鸭子匹配。
- `fit A for B` 中右侧 `B` 为普通 `type`，包括 `fit A for A`。
- 把 `fit` 解释成“任意类型之间的普遍桥接语法”。

## 7 结论

当前版本中，`spec` 与 `fit` 应视为同一设计面的两个部分:

- `spec` 负责声明契约。
- `fit` 负责声明适配，并保留类型自身的自扩展能力。

因此两者应放在同一份说明文件中；若后续文档继续拆分，也应保持“先说明 `spec`，再说明 `fit`”的顺序，而不应再把 `fit` 单独写成脱离 `spec` 的开放式机制文档。
