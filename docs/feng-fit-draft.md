新增 fit 关键字，本篇是 `fit` 关键字的完整技术方案总结。该设计旨在平衡 **鸭子类型的灵活性** 与 **强类型系统的严谨性**，通过“显式声明”与“自动边界收敛”解决工程实践中的重构与冲突难题。

---

# 语言设计规约：`fit` 协议

## 1. 设计核心哲学
* **默认全不开放 (Default-Deny)：** 即使两个类型结构完全一致，在没有显式声明的情况下，它们彼此保持隔离。
* **显式意图 (Explicit Intent)：** 所有的类型能力提升或关系建立，必须通过 `fit` 关键字进行“签字画押”。
* **非侵入性适配 (Non-Invasive)：** 无需修改原始类型的定义，即可为其扩展功能或适配接口。

---

## 2. 语法形态与用法

`fit` 具备三种进阶用法，统一了接口实现、扩展方法和适配器模式。

### A. 结构匹配 (Structural Bridging)
声明类型 `A` 符合契约 `B`（B 其实也是 Type）。
```feng
fit User for Named;
```

### B. 显式适配 (Adapter Pattern)
在建立关联的同时，补齐目标契约缺失的方法。
```feng
fit ThirdPartyUser for MyInterface {
  fn get_name(self): string { 
    return self.first + " " + self.last
  }
}
```

### C. 能力注入 (Extension Methods)
支持 `fit A for A`，可简写为 `fit A`，用于为类型注入新的方法。这些方法在满足孤儿规则的前提下，可以改变类型的“结构指纹”。
```feng
fit User {
  fn say_hi(self) {
    print("Hello, " + self.name)
  }
}
```

---

## 3. 作用域与可见性原则

### 包内共识 (Package-Level Consensus)
* **可见性对齐：** `fit` 的可见性规则与普通类型（Type）或函数（Fn）完全一致。
* **显式加载 (Self-Consistency)：** 跨模块（Module）使用时，必须显式 `use` 包含 `fit` 声明的模块。
* **逻辑自恰：** 只要开发者同时 `use` 了类型 $A$ 和相关的 `fit` 声明，该能力在当前作用域内即刻激活。

### 跨包边界 (Cross-Package Boundary)
* **导出规则：** 只有满足“非孤儿条件”的 `fit` 声明可以被标记为 `pu` 并导出。
* **孤儿规则 (Orphan Rules)：** * 若要 `pu fit A to B`，当前包必须是 $A$ 或 $B$ 其中之一的作者（即 $A$ 或 $B$ 在当前包内）。
    * 目的是防止第三方包滥用权力，导致全局生态中的类型关系冲突。

---

## 4. 编译器行为：自动作用域收敛

对于不满足孤儿规则但带有 `pu` 标记的声明，编译器采取 **“成功且收敛”** 的处理策略，而非报错或警告。

| 声明情况 | 编译器处理 | 最终可见性 | 开发者反馈 |
| :--- | :--- | :--- | :--- |
| **非孤儿声明** | 正常处理 `pu` 属性 | 全局可导出 | 编译成功 |
| **孤儿声明** | **自动收敛作用域** | **仅包内可见** | **[Success]** 已成功将孤儿声明限制在包内 |

> **设计考量：** 这种语义转换将“禁止”变为“保护”。编译器承认开发者在包内进行局部适配的合理性，同时通过强制降级（Internal Only）确保了外部生态的零污染。

---

## 5. 核心原则总结

### 1. 所见即所得
代码顶部的 `use` 列表即为当前文件能力的全部来源。不存在隐式的、全局飘荡的类型转换。

### 2. 局部自治
开发者可以在自己的包（Package）内自由地通过 `fit` 改造任何第三方类型以适配自己的需求，而无需担心这些改动会影响到其他包。

### 3. 专业级安全感
编译器通过 **“肯定语义”** 引导开发者编写合规的代码。它不仅仅是代码的检查者，更是包边界的维护者，确保了大型工程在频繁重构时的绝对确定性。

---

**结语：** `fit` 不仅仅是一个关键字，它是对“如何优雅地在解耦与严谨之间取舍”的一种工程回应。它让开发者在享受鸭子类型便利的同时，拥有了足以应对数十万行代码规模的重构底气。
