# Feng 语言 GC 与内存模型规范

本文档用于补充 [feng-language.md](./feng-language.md) 中的 GC 概要说明,聚焦 Feng 语言中的托管对象范围、可达性规则、C 互操作边界与资源管理约束。

> **设计原则基础**: 普通 `type` 是托管对象,运行时最终可采用 RC、GC 或等价策略; `@fixed` ABI 值不受该机制管理。详见 [Feng 语言设计原则](./feng-principles.md)。

## 1 GC 模型概览

- Feng 对托管对象采用自动内存管理语义,语言层只规定对象存活语义,不限定最终运行时采用 GC、RC 还是等价策略。
- 运行时通过可达性判断对象是否存活,而不是通过手动释放。
- 自动内存管理机制负责 Feng 原生对象图; `@fixed` ABI 值、C 侧对象与原始内存不由该机制接管。

## 2 托管对象与非托管对象

### 2.1 GC 托管对象

- `string`
- 动态数组
- Feng 原生 `type` 对象
- 闭包对象及其捕获环境

### 2.2 非托管对象

- `@fixed type` 与 `@fixed @union type` 定义的 ABI 兼容值
- `*T` 指针
- 由 C 库分配的原始内存
- 文件句柄、套接字、数据库连接等外部资源本体

## 3 可达性与根集合

以下位置中的对象引用构成 GC 根集合:

- 当前调用栈上的局部变量与参数
- 模块级 `let` / `var` 变量
- 当前求值过程中的临时值
- 闭包捕获环境中仍可访问的绑定

规则说明:

- 只要对象从根集合可达,该对象及其可达子对象都必须保持存活。
- 模块级变量在程序运行期间始终作为根保留。
- 闭包若捕获外部对象引用,则被捕获对象会随闭包一同延长生命周期。
- GC 必须正确处理循环引用。

## 4 与闭包和对象图的关系

- 闭包可捕获外部作用域绑定,被捕获的绑定会被提升为可被 GC 跟踪的运行时对象。
- `type` 对象之间可以相互引用,数组和对象也可以形成环。
- 程序员无需手动断开对象环来释放内存。

## 5 C 互操作边界

- 托管对象不得直接出现在 `@fixed fn` 的参数、返回值、`@fixed type` 成员或 `@fixed` 函数类型签名中。
- 需要与 C 交换数据时,必须通过 `@fixed` 兼容类型、`extern fn` 导入声明允许的 C 兼容表示或其他约定的中间表示完成。
- 当前语言版本不提供内建 pin、borrow 或裸引用导出机制。
- 将 GC 托管对象暴露给 C 侧长期持有属于未定义行为,语言层面禁止此类接口。

## 6 资源释放与终结规则

- Feng 当前不提供用户自定义析构函数或终结器。
- GC 只负责回收 Feng 运行时对象内存,不自动关闭外部资源。
- 持有外部资源的 Feng 对象应提供显式的 `close`、`free` 或同类释放函数。
- 若外部资源封装在 `type` 中,程序员仍需按协议主动释放资源,不能依赖 GC 的回收时机。

## 7 异常与 GC 的关系

- 异常传播不会破坏 GC 正确性; 进入异常展开阶段时,仍然可达的对象必须保持存活。
- `finally` 中执行的清理逻辑只负责业务资源收尾,不改变 GC 的可达性判定模型。
- 未捕获异常导致程序终止时,运行时不保证按业务语义执行外部资源回收。
