# Feng 语言设计原则

本文档用于沉淀 Feng 语言层面的设计原则,说明为什么语言规则被设计成当前形式,并为具体规范文档提供统一的判断依据。若具体规则文档与本文档表达的原则出现冲突,应优先修正规则文档,而不是在规则层继续堆积例外。

## 1 分层原则

- 关键字决定语法类别。凡属于语法结构的问题,都应能在 parse 阶段被检查并报错。
- 注解只影响语义分析与代码生成,不改变语法产生式,也不决定某条语句在 parse 阶段是否合法。
- `extern fn` 保留为关键字语法,因为它表达的是“外部 C 函数声明、没有 Feng 实现、不能有函数体”这一语法类别。
- `@fixed` 保留为内建注解,因为它表达的是“声明希望进入 ABI 固定边界”这一语义资格,而不是新的语法类别。
- 未来若引入自定义注解,也应遵循同一分层原则: 自定义注解可以增加语义和代码生成约束,但不应改写核心语法。

## 2 C互操作原则

- `extern fn` 仅用于声明 C 语言实现的外部函数。
- `@fixed` 用于标记 C ABI 固定属性,可作用于 `type`、`fn` 和方法声明。
- 对象形式的 `@fixed type` 用于声明 C 兼容结构体; `@fixed @union type` 用于声明 C 兼容联合体。
- 可调用形状的 `@fixed spec` 用于声明 C 兼容函数指针类型; 对象形状的 `spec` 不得标记 `@fixed`。
- 带参数的 `@cdecl`、`@stdcall` 和 `@fastcall` 只用于无函数体 `extern fn` 的导入声明; 无参数形式只用于 `@fixed fn` 或 `@fixed` 方法的 ABI 调用方式。
- 哪些声明与类型能够进入 C ABI 边界,以及对应的兼容资格与桥接规则,统一见 [Feng 语言 C 互操作规范](./feng-interop.md)。

## 3 ABI 资格原则

- `@fixed` 的合法性属于语义资格检查。某个声明是否能标记为 `@fixed`,由其直接依赖的类型语义决定,而不是由 `@fixed` 改写后的特殊语法决定。
- `@fixed` 声明的直接字段类型与可调用 ABI 边界上的参数/返回值兼容资格,统一见 [Feng 语言 C 互操作规范](./feng-interop.md)。
- 某个具名类型只有在其自身也通过 `@fixed` 合法性校验后,才属于 ABI 稳定类型集合。
- 普通 `type` 可以包含 `@fixed` 值字段; 但 `@fixed type` 的直接布局不能直接依赖普通 `type`、对象形状 `spec`、`string`、数组、闭包或其他托管对象类型。
- `@fixed` 资格的传播由语义分析沿类型依赖关系完成; 规范层只要求直接依赖满足资格,不必把实现细节写成递归语法。

## 4 布局与代码生成原则

- `@fixed type` 的短中期实现目标是直接映射到 C `struct`。
- `@fixed @union type` 的短中期实现目标是直接映射到 C `union`。
- `@fixed type` 的实例布局仅由字段决定; 方法、构造函数、访问控制和注解本身都不参与实例内存布局计算。
- 编译器不得为 `@fixed type` 或 `@fixed @union type` 隐式注入对象头、虚表指针、判别标签或其他隐藏实例字段。
- `@fixed type` 中可以存在普通方法,但普通方法不自动进入 C ABI 边界; 方法只有在其自身也标记为 `@fixed` 时,才按 ABI 兼容函数检查。
- 为了保持可直接编译为 C 的目标,`@fixed` 相关代码生成应优先采用可预测、可直接映射为 C 声明和 C 函数的策略,避免引入需要额外运行时对象头或隐藏上下文的机制。

## 5 生命周期原则

- 普通 `type` 是托管对象，由运行时自动内存管理机制负责生命周期，不影响语言层语义。
- `@fixed` 只额外约束进入 C ABI 边界时的布局与资格,不单独引入另一套 Feng 侧生命周期、比较或初始化语义。
- 值是否随作用域失效、是否需要显式释放,由其存储位置和外部资源拥有关系决定,不能仅因标注了 `@fixed` 就作不同语义推断。
- 若某个值内部持有外部资源,资源释放协议应由显式函数或外部约定完成,不能依赖注解本身替代协议。

## 6 诊断原则

- `extern fn` 的错误属于语法错误,应描述为“`extern fn` 不能有函数体”之类的结构性问题。
- `@fixed` 的错误属于语义资格错误,应描述为“该声明不能标记为 `@fixed`”,而不是“`@fixed` 改变了这种语法可写的成员种类”。
- 诊断信息应尽量指出不满足资格的具体原因,例如字段、参数或返回类型中存在非 ABI 稳定类型。

推荐诊断风格示例:

- `type PointList` 含有非 `@fixed` 的成员 `items`,因此不能标记为 `@fixed`。
- `fn point_sum` 的参数 `user` 类型不是 ABI 稳定类型,因此该函数不能标记为 `@fixed`。
- `method Buffer.write` 的返回类型不满足 `@fixed` 约束,因此该方法不能标记为 `@fixed`。
