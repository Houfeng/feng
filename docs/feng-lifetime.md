# Feng 语言对象生命周期规范

本文档补充 [feng-language.md](./feng-language.md) 中的内存管理概要，规定 Feng 语言采用自动引用计数（ARC）作为托管对象的生命周期管理机制，并配合运行时循环检测器回收不可达的循环引用组，聚焦托管对象范围、引用计数语义、C 互操作边界与资源管理约束。

## 1 职责

- 以 ARC 机制管理 Feng 原生对象的生命周期，使对象存活期精确对应其强引用存在期。
- 配合运行时循环检测器，对不可从根集合到达的循环引用对象组进行回收，确保所有不可达托管对象最终得到释放。
- 规定托管对象与非托管对象的边界，以及 ARC 在 C 互操作边界上的行为约束。

## 2 术语

- 强引用（strong reference）：持有对象所有权的引用；每存在一个强引用，对象引用计数加一；强引用归零时对象被立即释放。
- 引用计数（reference count）：对象当前被持有的强引用数量；由 ARC 在每次赋值、传参与返回时自动维护。
- 托管对象（managed object）：生命周期由 ARC 机制负责的 Feng 原生对象，详见 §3.1。
- 非托管对象（unmanaged object）：生命周期不受 ARC 控制的值或外部资源，详见 §3.2。
- 循环引用（retain cycle）：两个或多个托管对象之间形成的强引用环；环内对象的引用计数均不为零，导致 ARC 无法自动释放；当整个环对外部根集合不可达时，由运行时循环检测器识别并回收。
- 循环检测器（cycle collector）：运行时中负责识别并回收不可达循环引用组的组件；不干预正常 ARC 的确定性释放语义，仅处理 ARC 单独无法回收的不可达环。
- 释放（release）：对象强引用计数降为零时，ARC 立即销毁该对象并回收其内存。
- 终结器（finalizer）：对象在被 ARC 或循环检测器确认释放之前，由运行时自动调用的用户定义清理函数；其声明形式与约束见 [Feng 语言类型规范](./feng-type.md)，执行流程见 §13.2。

## 3 托管对象范围

### 3.1 ARC 托管对象

以下类型的值由 ARC 管理：

- `string`
- 数组（`T[]` 及多维形式）
- Feng 原生 `type` 对象实例
- 闭包对象及其捕获环境

### 3.2 非托管对象

以下类型的值不受 ARC 管理：

- `@fixed type` 与 `@fixed @union type` 定义的 ABI 居靠地址存储的对象：在 Feng 内部采用引用语义，但编译器不为其对象头注入引用计数，不执行 retain/release；生命周期由程序员通过显式方法将管理。
- `*T` 指针
- 由 C 库分配的原始内存
- 文件句柄、套接字、数据库连接等外部资源本体

## 4 ARC 语义

- 对托管对象的引用默认为强引用；赋值、传参、返回与闭包捕获均自动 retain，绑定或临时值生命周期结束时自动 release。
- 同一对象可被多个变量同时强引用；最后一个强引用消失时，ARC 执行终结器（若已定义），终结器完成且引用计数仍为零时立即释放对象内存；未定义终结器的对象在强引用归零时即刻被释放，无暂停、无延迟。
- ARC 的 retain/release 操作由编译器在赋值点自动插入，程序员无需手动管理。
- 模块级 `let` / `var` 变量在程序运行期间始终持有强引用，对应对象在程序退出前不被释放。
- 当前调用栈上的局部变量与参数、当前求值过程中的临时值，均持有强引用；离开作用域或表达式结束时自动 release。
- 闭包对外部作用域绑定的捕获是引用捕获：被捕获的绑定提升为由 ARC 跟踪的运行时对象；捕获 `var` 绑定时，闭包与外层作用域共享同一存储，闭包内对该绑定的赋值对外层及其他捕获了同一绑定的闭包均可见；捕获 `let` 绑定时，该绑定在闭包内仍不可重新赋值。
- Feng 不提供 `weak` 或 `unowned` 引用语法；所有托管引用均为强引用；循环引用的回收由运行时循环检测器统一处理，无需程序员显式标注弱引用。不引入显式弱引用原语，有助于将语言层语义与具体内存管理实现解耦，为未来切换内存管理后端或编译至多目标提供平滑路径。

## 5 循环引用

- `type` 对象之间可以相互强引用，也可形成与数组、闭包混合的对象图；若图中存在强引用环，则环内对象的引用计数永远不会归零，构成循环引用，ARC 单独无法回收这些对象。
- 循环引用由运行时循环检测器负责处理：当一组循环引用对象对外部根集合完全不可达时，循环检测器识别该对象组并回收其内存；程序员无需为循环引用做任何额外处理。
- 循环检测器仅处理对外不可达的环；仍被根集合强引用的对象路径不指向任何环的对象，不受循环检测器影响。
- 循环检测器采用 Bacon-Rajan 算法方向：`release` 时引用计数降低但未归零的 potentially-cyclic 对象加入候选缓冲区，定期对缓冲区做试验性删除扫描以识别不可达环；触发策略与实现细节见 §13.1。

## 6 与闭包和对象图的关系

- 闭包捕获外部对象引用时，被捕获对象的引用计数增加；闭包本身被释放时，对应引用计数随之减少。
- `type` 对象之间可以相互引用，数组与对象、闭包也可以组合为任意形状的对象图；无论对象图是否包含环，对外不可达的对象最终都会被回收。

## 7 C 互操作边界

- `string` 与数组可按 [Feng 内建类型规范](./feng-builtin-type.md) 中定义的桥接规则出现在 `extern fn`、`@fixed fn`、`@fixed` 方法与 `@fixed spec` 的参数和返回值位置。
- 除 `string` 与数组这类桥接类型外，其他托管对象不得直接出现在 `@fixed fn` 的参数、返回值位置，也不得出现在 `@fixed` 函数类型签名中；托管对象同样不得成为 `@fixed type` 的直接成员。
- 需要与 C 交换数据时，可直接使用 [Feng 语言 C 互操作规范](./feng-interop.md) 允许的 ABI 稳定类型或 ABI 桥接类型；开发者无需手写字符串或数组的中间表示。
- ARC 的 retain/release 语义可按约定暴露给 C 侧：C 代码通过调用约定的 retain/release 函数管理对 Feng 对象的引用，不需要 barrier 机制。
- C 代码长期持有 Feng 对象引用时，必须在持有期间保持相应的 retain；未按约定 retain 而直接持有 Feng 对象引用属于未定义行为。
- 当前语言版本不提供内建 pin、borrow 或裸引用导出机制。

## 8 资源管理

- ARC 负责回收 Feng 托管对象的内存，不自动关闭文件句柄、套接字、数据库连接等外部资源。
- `type` 对象可定义终结器；终结器的声明规则见 [Feng 语言类型规范](./feng-type.md)，执行流程与释放规则见 §13.2。
- 持有外部资源的 `type` 对象应提供显式的 `close`、`free` 或同类释放方法；程序员需按协议主动调用，不可依赖 ARC 的释放时机来触发外部资源回收。
- 外部资源应在明确的代码位置显式释放，以确保释放时机可预期。

## 9 异常与生命周期

- 异常传播不破坏 ARC 正确性；异常展开阶段中，仍持有强引用的对象保持存活。
- 异常展开过程中，离开作用域的局部变量会正常触发 release；ARC 的 retain/release 计数在异常路径与正常路径上语义一致。
- `finally` 中执行的清理逻辑负责业务资源收尾，不影响 ARC 的引用计数模型。
- 未捕获异常导致程序终止时，运行时不保证按业务语义执行外部资源回收。

## 10 规则

分为「必须、禁止」。

- [必须] 托管对象的生命周期由 ARC 管理；程序员不得手动释放 ARC 托管对象。
- [必须] `string` 与数组可按 [Feng 内建类型规范](./feng-builtin-type.md) 中定义的桥接规则出现在可调用 C ABI 边界的参数与返回值位置。
- [必须] 除 `string` 与数组这类桥接类型外，其他托管对象不得直接出现在 `@fixed fn` 的参数或返回值位置，也不得作为 `@fixed type` 的直接成员。
- [必须] C 侧持有 Feng 托管对象引用时，必须按约定调用 retain/release 以维护引用计数正确性。
- [必须] 持有外部资源的对象必须提供显式释放方法，调用方须负责主动调用。
- [禁止] 将 ARC 托管对象暴露给 C 侧长期持有而不遵循 retain/release 约定。

## 11 编译期

- 编译器必须检查托管对象是否出现在 `@fixed` 边界的允许或禁止位置：`string` 与数组可按 [Feng 内建类型规范](./feng-builtin-type.md) 中定义的桥接规则出现在可调用 ABI 边界上；其他托管对象若出现在禁止位置则必须报错。
- 编译器对全部托管类型构造类型引用图（节点为托管类型，有向边为"类型 A 的字段中包含类型 B 的托管引用"），在该图上求强连通分量（SCC）；落于非平凡 SCC 中的类型标记为 **potentially-cyclic**，其余标记为 **acyclic**；分析基于 `.fi` 字段类型信息，可跨包进行，不需要全程序分析；编译器对 potentially-cyclic 类型结构可发出信息性提示（不影响编译结果）。
- 编译器为每个托管 `type` 生成引用枚举元数据（托管字段的偏移与类型描述符列表），供运行时循环检测器遍历对象引用时使用。
- 编译器自动插入 retain/release 操作，不需要程序员手动标注。

## 12 运行时

- ARC 维护每个托管对象的引用计数；计数降为零时立即销毁该对象并回收内存，释放是确定性的、非延迟的；引入终结器后参见 §13.2 中的释放流程。
- 循环检测器在 ARC 之外单独运行，仅处理 potentially-cyclic 类型对象构成的不可达循环引用组；acyclic 类型对象不进入候选缓冲区，ARC 路径上零额外开销。
- 循环检测器采用 Bacon-Rajan 方向：`release` 时引用计数降低但未归零的 potentially-cyclic 对象加入候选缓冲区；定期对缓冲区执行试验性删除扫描，将确认不可达的对象组回收；若组内对象定义了终结器，则执行两阶段收集流程（见 §13.2）。
- `string`、数组、普通 `type` 对象实例与闭包环境均由 ARC（含循环检测器）管理；`@fixed type` 对象、`*T` 指针与 C 侧分配内存不受该机制管理。
- 触发策略、多线程原子性策略与终结器执行流程见 §13。

### 12.1 `string` 运行时表示

当前运行时中，`string` 表示为受管堆对象；语言层 `string` 值是指向该对象的强引用。对象布局等价于：

```c
typedef struct FengString {
   FengManagedHeader header;
   size_t len;
   uint8_t data[];
} FengString;
```

其中：

- `len` 表示字符串内容的 UTF-8 字节长度。
- `data[0..len)` 为字符串内容。
- 实际分配时至少额外保留 `1` 个结尾 `0` 字节，供运行时内部与只读 C 桥接使用；该字节不属于字符串内容，也不计入 `len`。
- 字符串对象创建完成后内容不可原地修改；任何结果不同的字符串操作都生成新的 `string` 值。

## 13 待定事项

### 13.1 循环检测器算法与触发策略

- **算法方向**：Bacon-Rajan。`release` 时若对象引用计数降低但未归零，且该对象为 potentially-cyclic 类型，则将其加入候选缓冲区；定期对缓冲区执行试验性删除扫描（对候选对象的可达引用计数做试验性递减，检查是否归零）以识别不可达环，确认后回收。
- **触发策略**：尚未确定，候选方案包括固定频率周期运行、按分配或 release 次数阈值触发等。
- **可观察延迟**：循环检测器的回收可能引入延迟；环内对象的释放时机不具备确定性；acyclic 类型对象的释放路径不经过循环检测器，仍为确定性。
- **多线程原子性**：多线程场景下原子引用计数与非原子引用计数的策略选择尚未确定，影响线程模型与性能特征。

### 13.2 终结器执行流程

`type` 对象可定义终结器；终结器的定义形式与语义约束见 [Feng 语言类型规范](./feng-type.md)；以下仅说明生命周期与执行流程。

**ARC 路径释放流程（有终结器时）：**

1. 对象强引用计数降为零；
2. ARC 执行该对象的终结器；
3. 终结器执行完毕后重新检查引用计数：
   - 若引用计数 > 0（对象在终结器内被复活）：对象恢复存活，本次不释放；下次引用计数再次降为零时，终结器将再次执行；
   - 若引用计数仍为零：ARC 立即释放对象内存。

**循环检测器路径释放流程（有终结器时）：**

循环检测器确认一组对象对外不可达后，若组内存在定义了终结器的对象，采用两阶段收集：

- **Phase 1**：对组内所有有终结器的对象各执行一次终结器，执行顺序不保障；所有终结器执行完成后进入 Phase 1.5。Phase 1 期间组内所有对象的内存均有效，跨对象字段访问不会产生 use-after-free。
- **Phase 1.5（存活 re-check）**：扫描组内每个对象的实际引用计数；若某对象的实际引用计数高于其组内内部引用计数（即组外建立了新引用，对象被复活），则从该对象出发，沿强引用出边在组内做 BFS 传播存活，将所有可达的组内对象一并标记为存活；未被标记为存活的对象进入 free_set。
- **Phase 2**：释放 free_set 中所有对象的内存。

被复活的对象下次再次不可达时，将再次经历上述流程，终结器将再次执行。

**不推荐的终结器用法（不影响内存安全，但可能导致资源泄漏或逻辑错误）：**

- 在终结器内复活对象（将 `self` 或同组内其他对象赋给全局变量或长期存活的对象）：若每次 finalize 均复活，对象将永久不被释放，资源永久泄漏。
- 在终结器内访问同一收集组内其他对象的字段：对方的终结器可能已执行，字段语义不可信。
- 依赖循环组内终结器的执行顺序：顺序不保障。

**设计约束（规划）：**

- 终结器不得抛出异常；终结器内的可能抛出操作须由程序员自行 `try`/`catch` 处理。若异常未被捕获而传播到终结器边界，运行时直接 crash，不尝试恢复或继续执行同组内其他终结器。
- 终结器为运行时自动调用的特殊成员函数，不可由程序员直接调用。

### 13.3 C retain/release 接口规范

C 代码持有 Feng 对象时，retain/release 约定的详细接口文档与规范尚未制定。
