# Feng 内存管理草案：引用计数（RC）

本文档整理 feng 语言内存管理方案的设计分析与决策依据，尚未最终定论。

---

## 1 GC vs RC 的权衡

### 1.1 GC 的问题

- **Stop-the-world 暂停**：延迟不可预测，实时/嵌入式场景不可接受
- **C 互操作复杂**：GC 移动对象时需要 pinning 机制；C 持有 feng 对象指针期间必须通知 GC，否则对象被回收。这与 feng 以 C 互操作为一等公民的设计目标冲突
- **运行时重量**：需要独立的 GC 运行时，不适合 CLI 工具或小体积二进制

### 1.2 RC 的优势

- **延迟确定**：对象释放时机精确，析构顺序可预测
- **C 互操作自然**：`retain`/`release` 语义可直接暴露给 C，C 代码通过约定调用即可，不需要 GC 感知。Swift、CPython、ObjC 均采用此方案，实践成熟
- **运行时轻量**：不需要独立扫描线程，适合嵌入和 CLI

### 1.3 RC 的已知问题：循环引用

RC 无法自动回收强引用环。传统解决方案是：

- 语言层提供 `weak` / `unowned` 引用原语，由用户显式标注打破循环
- 编译器提供静态 retain cycle 警告，提前发现问题

循环引用静态检测能否实现，取决于编译器是否具备完整的类型图。

## 2 feng 类型系统对静态检测的支持

### 2.1 编译期结构类型

feng 是**编译期结构类型（structural typing）**，而非运行时鸭子类型。所有类型匹配在编译期严格完成，不推迟到运行时。这意味着：

- 每个对象字段在编译期都有确定的结构类型
- 编译器在每个赋值点均已解析具体类型
- 编译器可建立完整的具体类型图

### 2.2 `.fi` 中的完整类型信息

源码中的类型推断在编译后全部固化为显式类型写入 `.fi`：

```feng
// 源码
pu type Foo {
    let x = 1;
    let y: string;
}
```

```feng
// 编译后写入 .fi
pu type Foo {
    @bounded
    let x: number;
    let y: string;
}
```

`@bounded` 记录值绑定，类型信息比普通类型注解更丰富。任何消费 `.fi` 的编译单元看到的都是**完整、无歧义的类型图**。

### 2.3 对循环引用检测的含义

- 跨包类型图也是可分析的：编译器扫描依赖包的 `.fi` 即可得到完整字段类型
- 静态 retain cycle 检测**不需要全程序分析**，只需跟踪字段类型链直到发现强引用环
- `weak` 标注可在 `.fi` 层面直接表达，消费方可见

feng 完整固化的 `.fi` 类型信息，使得静态循环引用检测比很多有名义类型系统的语言更容易实现。

## 3 结论

| 维度 | GC | RC（feng 选择）|
|------|----|----|
| C 互操作难度 | 高（需 pinning/barrier）| 低（retain/release 约定）|
| 延迟确定性 | 差 | 好 |
| 运行时体积 | 大 | 小 |
| 循环引用 | 自动处理 | 需 `weak` 标注 |
| 静态循环检测 | N/A | **可行**（依赖 `.fi` 完整类型图）|

RC 与 feng 的设计目标高度吻合：

1. C 互操作是一等公民 → RC 在 C 边界无额外机制
2. 编译期结构类型 + `.fi` 完整类型信息 → 静态 retain cycle 警告可实现
3. 轻量运行时 → RC 不引入独立扫描线程
