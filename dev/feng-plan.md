## 1. 目标

本文档用于收敛 Feng 当前实现阶段的整体路线，明确发码、运行时、CLI、包系统、C ABI 与标准库的实施顺序。

当前推荐路线：

- 第一代后端优先选择 C，而不是 LLVM。
- 本地源码优先，先打通本地源码到可执行文件的闭环。
- C ABI 在前几个阶段保持现状，不提前扩大兼容面。
- ARC 生命周期管理先交付可用核心，再补高风险能力。

这样可以尽快进入“本地源码 -> 生成目标代码 -> 宿主编译器 -> 可执行文件”的可验证闭环，避免把后端、运行时、包系统、C ABI 细节一次性耦合到同一个阶段里。

## 2. 总体决策

### 2.1 后端选择

第一代后端优先选择 C。

原因：

- 现有规范已经明确 Feng 维持 feng -> C 的总体架构。
- 当前仓库只有 C 实现的前端与 C 构建链，没有 LLVM、IR、SSA、中端优化和后端基础设施。
- C 后端更贴合当前 `@fixed`、`extern fn`、包产物与头文件生成方向。

LLVM 不是禁止项，但更适合作为 C 后端稳定之后的新增后端，而不是第一代后端的起点。

### 2.2 ARC 第一阶段拆分

虽然目标上可以称为“第一阶段实现 ARC 生命周期管理”，但实施上建议拆成两个子里程碑：

- 1A：确定性 ARC 核心
- 1B：循环回收与复杂终结器语义补完

如果把 cycle collector、终结器复活、复杂异常交互、C retain/release 一并塞进第一波交付，阶段 1 的失败面会显著增大。

### 2.3 本地源码优先

第二阶段不建议只停在“产出目标代码文件”，而建议直接做到“本地源码 -> 可执行文件”。

否则 CLI、运行时、发码接口和宿主编译器对接都会被推迟，后面会集中爆炸。

## 3. 阶段总览

### Phase 1A：后端与运行时骨架

目标：

- 建立 C 发码主链路。
- 建立最小运行时骨架。

主要工作：

- 基于现有 parser/semantic 输出新增后端驱动。
- 新增 C 发码模块。
- 新增运行时公共头与最小运行时库。
- 明确代码生成输入直接使用语义分析后的 AST 与附着的 resolved callable / inferred type 元数据，不重新做一次解析。

建议产物：

- `src/codegen/` 或等价目录
- `src/runtime/` 或等价目录
- 最小 smoke test 驱动

### Phase 1A：确定性 ARC 核心

目标：

- 先交付可用的 ARC 基础能力。

覆盖范围：

- `string`
- 数组
- 普通 `type`
- 闭包环境

核心能力：

- 托管对象头
- 原子 retain / release
- 构造/析构入口
- 作用域退出清理
- 异常路径清理
- 单路径终结器执行

第一版终结器建议：

- 引用计数归零
- 调终结器
- 终结器不承诺固定执行线程，由运行时根据实际情况决定
- 若未复活则立即释放

此阶段先不强求：

- cycle collector
- 终结器复活完整语义
- 多阶段回收
- C retain/release 接口

依赖：

- 依赖后端与运行时骨架。

### Phase 1A：本地源码 C 发码最小闭环

目标：

- 打通本地源码到 C 代码的最小闭环。

优先覆盖语义面：

- 模块级 `let` / `var`
- 顶层 `fn`
- 普通 `type`
- 构造函数
- 方法调用
- 对象字面量
- 数组
- 控制流
- 异常展开
- `main` 入口

输出要求：

- 生成 C 文件
- 能引用最小运行时

依赖：

- 依赖运行时骨架与确定性 ARC 核心。

### Phase 1A 验证里程碑

硬门槛：

- `examples` 中的本地源码可编译成 C。
- 生成的 C 能通过宿主 C 编译器生成可执行文件。
- 生成的可执行文件能够正确运行最小示例。

达到该门槛后，才进入下一阶段。

### Phase 1B：补完 ARC 高风险能力

目标：

- 补完确定性 ARC 之外的复杂能力。

范围：

- `potentially-cyclic` 类型对应的循环检测器
- 候选缓冲区
- 基础 cycle collector
- 终结器复活检查
- 复杂异常/终结器交互

建议：

- 该阶段推荐在本地闭环打通后并行推进。
- 不建议阻塞本地源码可执行闭环。

### Phase 2：支持本地源码直接构建到可执行文件

目标：

- 把当前调试型 CLI 演进为真正的单包本地编译驱动。

能力范围：

- 解析目标类型
- 指定产出路径
- 调用宿主 C 编译器
- 组织运行时与生成文件

边界：

- 此阶段仍不处理外部包。
- 此阶段不扩大 C ABI 兼容面。

建议：

- 推荐依赖 Phase 1A 闭环达成后推进。
- 不必等待 Phase 1B 全部完成。

### Phase 3：完善 CLI，支持构建 `.fb` 包

目标：

- 完善本地项目工作流。
- 支持本地项目构建与打包。

能力范围：

- `feng.fm` 解析
- 本地项目 `build`
- 本地项目 `check`
- 本地项目 `run`
- 本地项目 `clean`
- 本地项目 `pack`
- `.fb` 打包
- 本地包产物布局
- `.fi` 生成（作为本阶段最后一项功能任务）

边界：

- 此时只要求“本地项目能打包”。
- 不要求外部依赖解析。
- `.fi` 生成仍在 Phase 3 内完成，但安排在 `build/check/run/clean/pack` 与 `.fb` 打包稳定之后，作为阶段末项处理。

### Phase 4：支持外部包 + 本地源码

目标：

- 支持“本地源码 + 外部 `.fb` 包”混编。

能力范围：

- 包索引
- 本地缓存
- `mod/.fi` 解析
- `lib/clib` 选择
- `use` 的跨包解析

要求：

- 本地源码可以引用外部 `.fb`。
- 现有本地源码路径不回退。

### Phase 5：细化 C ABI 兼容与互操作

目标：

- 把 C ABI 相关能力从“先保持现状”推进到“成体系实现”。

范围：

- `extern fn`
- `@fixed`
- 调用约定
- 头文件生成
- `string` / 数组桥接
- 异常不得越过 ABI 边界
- C 侧长期持有对象的 retain / release 协议

说明：

- 这一阶段再扩大 ABI 兼容集合。
- 这一阶段再补完整的互操作测试矩阵。

### Phase 6：实现必需标准库

目标：

- 引入编译器和运行时真正依赖的最小标准库。
- 在此基础上逐步扩展面向用户的基础库。

原则：

- 标准库建立在已经稳定的运行时、构建和包机制之上。
- 不建议把标准库变成前几个阶段的临时补丁仓库。

## 4. 阶段依赖关系

按推荐顺序：

1. Phase 0
2. Phase 1A 后端与运行时骨架
3. Phase 1A 确定性 ARC 核心
4. Phase 1A 本地源码 C 发码最小闭环
5. Phase 1A 验证里程碑
6. Phase 1B ARC 高风险能力补完
7. Phase 2 本地源码直接构建到可执行文件
8. Phase 3 CLI 与 `.fb` 打包
9. Phase 4 外部包 + 本地源码
10. Phase 5 C ABI 兼容与互操作
11. Phase 6 标准库

说明：

- Phase 1B 推荐与 Phase 2 并行，而不是硬阻塞。
- Phase 3 先解决“本地项目能打包”，并把 `.fi` 生成放到该阶段最后处理；Phase 4 再解决“消费外部包”。

## 6. 验证策略

### 6.1 阶段 1A

- 现有前端回归持续通过。
- 新增运行时单测覆盖 retain / release。
- 覆盖作用域退出释放。
- 覆盖构造/终结器调用。
- 覆盖异常展开清理。

### 6.2 阶段 1A / 2 端到端验证

- 对 `examples/hello_world.ff` 做“Feng 源码 -> 生成 C -> 宿主 C 编译器 -> 可执行文件 -> 运行结果”验证。
- 至少再准备一个包含对象、数组、异常的程序做同类验证。

### 6.3 阶段 1B

- 增加循环引用测试。
- 增加终结器复活测试。
- 增加复杂 `finally` 交互测试。
- 明确哪些行为在 1A 不保证、在 1B 才保证。

### 6.4 阶段 3

- 对本地项目执行 `build` / `check` / `run` / `clean` / `pack` 的 CLI smoke tests。
- 检查生成的 `feng.fm`、`mod`、`lib`、`clib`、`include` 布局是否符合规范。

### 6.5 阶段 4

- 至少准备一个“本地 app + 一个本地生成 `.fb` 包”的集成测试。
- 验证 `use`、`.fi` 解析、链接选择和冲突诊断。

### 6.6 阶段 5

- 增加 C 宿主调用 Feng 导出函数测试。
- 增加 Feng 调用 `extern fn` 测试。
- 增加 `@fixed` 类型与桥接 `string` / 数组的互操作测试。

### 6.7 阶段 6

- 标准库先以编译器自举所需最小集合为准。
- 新增 API 必须附带单测、集成测试和示例。

## 7. 当前结论

- 推荐先做 C 后端，不推荐先做 LLVM。
- 推荐把第一阶段拆成 1A 和 1B，而不是把完整 ARC 一次做完。
- 第二阶段建议做到“本地源码 -> 可执行文件”。
- 第三阶段先解决“本地项目能打包”。
- 第四阶段再解决“消费外部包”。
- 第五阶段之前不主动扩大 C ABI 兼容集合，也不承诺完整的 `string` / 数组桥接协议。
- 第六阶段标准库建立在已经稳定的运行时、构建和包机制之上。
