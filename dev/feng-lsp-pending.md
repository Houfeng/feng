# Feng LSP 待开发方案

> 本文档用于收敛 Feng 首版 LSP 与文档注释支持的开发方案。
> 原则只有两条：
> 1. **尽可能不改现有代码**。
> 2. **如果必须改，改动必须架构合理、范围最小、原因明确**。

## 1. 目标

本方案要解决两件事：

- 支持 Feng 的**专用文档注释**，并能被 IDE/LSP 正确消费。
- 落地 Feng 的**完整 LSP**，但**不把本地 workspace `.ft` 当作前置条件**。

本方案明确采用以下边界：

- 当前项目本地 `build/obj/symbols/**/*.ft` 只是**可选缓存**。
- LSP **有缓存就用**，**没有缓存就直接做源码分析**。
- `.ft` 由谁生成、何时生成，与 LSP **没有必然强绑定关系**。
- 外部依赖包中的公开 `.ft` 仍然是跨包消费接口，和“当前项目本地缓存 `.ft`”不是同一问题。

## 2. 已确认决策

### 2.1 缓存边界

- 若当前项目存在合法的 `build/obj/symbols/**/*.ft`，LSP 可直接把它作为 workspace 缓存消费。
- 若本地 workspace `.ft` 不存在或失效，LSP 必须直接回退到源码分析。
- LSP 不关心 `.ft` 由谁生成、何时生成。
- “是否要为当前项目生成 workspace `.ft`”属于后续缓存优化议题，不属于首版 LSP 前置条件。

### 2.2 LSP 与 `.ft` 的关系

- LSP **不依赖**开发者手动先执行任何外部命令。
- LSP **不依赖**本地 workspace `.ft` 一定存在。
- 若存在有效 workspace `.ft`，LSP 可直接消费它。
- 若不存在或失效，LSP 必须直接回退到源码分析。
- “源码分析”必须复用现有 parser / semantic / imported-module 能力，不允许重新发明一套平行语义逻辑。

### 2.3 文档注释语法

文档注释采用专用语法：

- `/** */` 文档块注释

只有能按绑定规则归属到声明的 `/** */`，才属于有效文档注释。
parser 只保留这种有效文档注释，并把它绑定到对应声明。

普通注释保持原语义：

- `//`
- `/* */`

普通注释不再因“靠近声明”自动升级为文档注释。

## 3. 现状事实

### 3.1 当前可复用能力

以下能力已经存在，首版 LSP 应优先复用：

- `src/cli/frontend.h` / `src/cli/frontend.c`
  - 已有共享前端管线：读文件、parse、semantic analyze、导入包注入、诊断回调。
  - 该管线已经支持把 `analysis`、`sources`、`imported_module_cache` 返回给调用方。
- `src/semantic/semantic.h`
  - 已有 `feng_semantic_analyze_with_options(...)`，可注入 imported module query。
- `src/symbol/provider.h` / `src/symbol/imported_module.c`
  - 已有通过公开 `.ft` 为外部依赖构造导入模块查询视图的能力。
- `editors/feng-vscode/extension.js`
  - 现有扩展已经有 formatter 与 diagnostics 集成，可作为接入 LSP client 的起点。

### 3.2 当前已知限制

- 现有命令行诊断路径只走“检查输出”路径，不会返回 LSP 需要的查询对象。
- 现有共享前端默认从磁盘路径读源码，不直接支持“未保存 buffer 的内存文本覆盖”。
- 当前文档注释规范写法分散，且实现链路尚未完整落地。

## 4. 总体方案

### 4.1 正确性优先路径

首版 LSP 的正确性路径如下：

1. 若当前项目存在合法的 workspace `.ft`，LSP 可以直接读取并消费。
2. 若本地 `.ft` 缺失、失效，或当前文档存在未保存修改，LSP 直接回退到源码分析。
3. 回退到源码分析时，必须复用现有前端 / 语义分析能力，而不是调用外部命令做字符串级中转。

### 4.2 性能优化路径

本地 workspace `.ft` 属于性能优化路径，而不是正确性路径：

- 首版 LSP 可在“没有本地 `.ft`”的情况下完整工作。
- 本地 `.ft` 的生成策略可以后续独立优化。
- 首版不规定 `.ft` 的生产者与生成时机。
- 只要存在合法缓存，LSP 即可直接消费。

## 5. 最小改动原则

首版开发必须遵守以下约束：

### 5.1 禁止面向 LSP 改动的部分

以下内容在首版 LSP 中**禁止因为 LSP 需求而改动**：

- `src/codegen/`
  - 原因：LSP 与 codegen 无直接前置关系，改动只会扩大耦合面。
- `src/runtime/`
  - 原因：LSP 不应拖动运行时层，改动会破坏职责边界。
- `src/semantic/`
  - 原因：现有语义分析能力已经属于复用对象；文档注释绑定应停留在 parser，LSP 也不应为此把语义层变成文档处理层。

### 5.2 不应前置重构的部分

以下内容**不应在首版 LSP 中作为前置重构目标**：

- 本地 workspace `.ft` 的生成策略
  - 原因：LSP 首版不能把它当作前置条件，也不应先绑定某个特定生产者。

### 5.3 必须坚持的复用方向

以下能力必须复用，不应复制实现：

- parser
- semantic analyzer
- imported module query / provider
- 现有 diagnostics 输出逻辑中的语义结果结构

## 6. 文档改动边界

文档层需要同步收敛三类事实，但本方案**不细列具体要改哪些 `*.md` 文件**：

- 文档注释的单一语法与绑定规则
- `feng lsp` 的命令表面
- `.ft` 对 LSP 只是可选缓存而不是前置条件

文档层要求只有两点：

- 最终表述必须一致
- 不再把“workspace `.ft` 由谁生成”写成 LSP 的必备前提

## 7. 必须改动的代码

以下改动是首版 LSP / 文档注释支持中，**架构合理且最小必要**的部分。

### 7.0 核心编译器改动边界

按当前方案，**核心编译器默认目标是尽量 0 改动**。

如果核心编译器因为 LSP 需要发生改动，必须满足两个条件：

- 改动点要单独列出
- 每个改动点都必须明确“为什么非改不可”

当前方案中，真正触碰核心编译器的改动只允许有两类：

1. 文档注释的词法 / 语法承载
2. 文档注释在 `.ft` / 符号视图中的文本文档载荷

除这两类之外，本方案**不预设**其他核心编译器改动为首版前置条件。
其中 `src/codegen/`、`src/runtime/` 与 `src/semantic/` 明确不在允许改动范围内。

### 7.1 顶层 LSP 命令路由

**需要改动的文件**：

- `src/cli/main.c`
- `src/cli/cli.h`

**为什么要改**：

- 需要给 `feng lsp` 一个正式顶层入口。
- 这是命令表面变更，无法绕开。

**最小改动策略**：

- 只新增 `lsp` 路由与声明。
- 不挪动现有其他顶层命令路由语义。

### 7.2 新增 LSP 服务端模块

**建议新增目录**：

- `src/cli/lsp/` 或等价目录

建议拆分：

- `main` / 路由入口
- JSON-RPC 读写
- session / document store
- request handlers

**为什么要改**：

- LSP 是新增能力，不应把协议处理逻辑塞回 `main.c` 或 `extension.js`。

**最小改动策略**：

- 以新增文件为主。
- 不把现有 CLI 公共逻辑大规模搬家。

### 7.3 共享前端增加“内存文本覆盖”能力

**需要改动的文件**：

- `src/cli/frontend.h`
- `src/cli/frontend.c`

**为什么要改**：

- LSP 必须支持未保存文件。
- 现有前端默认按路径从磁盘读文件，不够用。

**最小改动策略**：

- 保留现有 `feng_cli_frontend_run(...)` 入口不变。
- 新增一个支持 source overlay / in-memory source 的并行入口，供 `feng lsp` 使用。
- overlay 必须按源文件路径精确匹配；命中 overlay 时只替换源码文本，不改路径语义。
- 避免改动现有命令行调用方。

### 7.4 文档注释词法与语法承载

**需要改动的文件**：

- `src/lexer/lexer.c`
- `src/lexer/token.h`（若需要 trivia/注释载体）
- `src/parser/parser.h`
- parser 实现文件

**为什么要改**：

- 文档注释是语言事实，必须在 AST 上有承载位置。
- 如果 parser 不保留并绑定有效文档注释，hover 就无法稳定展示。

**最小改动策略**：

- parser 只保留可合法绑定到声明的 `/** */` 文档注释。
- parser 只负责把有效文档注释绑定到对应声明。
- 只增加文档注释承载字段与声明绑定所需最小数据结构。
- 无法绑定到声明的注释，不作为文档注释保留。
- 普通注释继续按原路径跳过。
- 不在首版引入复杂注释模型。
- 不把文档清洗、摘要提取或其他 LSP 专用逻辑塞进 parser。

### 7.5 文档注释导出到符号视图

**需要改动的文件**：

- `src/symbol/internal.h`
- `src/symbol/export.c`
- `src/symbol/ft_internal.h`
- `src/symbol/ft_write.c`
- `src/symbol/ft_read.c`
- `src/symbol/provider.h` 及对应实现

**为什么要改**：

- 若存在 `.ft`，LSP 读取缓存时也要能拿到文档注释。
- 当前只有 `has_doc`，没有真正的文本文档载荷。

**最小改动策略**：

- 只补齐 `DOCS` section 的真实读写与 getter。
- 不把局部变量、表达式级语义结果塞进 `.ft`。

### 7.6 VS Code 扩展切换为 LSP Client

**需要改动的文件**：

- `editors/feng-vscode/package.json`
- `editors/feng-vscode/extension.js`

**为什么要改**：

- 扩展需要启动 `feng lsp` 并接入标准 Language Client。

**最小改动策略**：

- 尽量保留现有 formatter 与 TextMate grammar 逻辑。
- 只把 diagnostics / completion / hover / definition 等语言能力切换到 LSP。

## 8. 不应在首版前置做的改动

以下内容可做，但**不应阻塞首版 LSP**：

### 8.1 先固定 workspace `.ft` 的生产者与生成时机

**状态**：允许后续做，但不前置。

**原因**：

- 这是性能优化，不是正确性前置条件。
- 先做会把 LSP 过早绑定到特定缓存生产策略上。

### 8.2 要求 LSP 自己先写 `.ft` 再服务请求

**状态**：不建议。

**原因**：

- 会让缓存重新变成前置条件。
- 会拖慢首个 request 响应路径。

### 8.3 用外部命令中转代替源码回退

**状态**：不建议作为主路径。

**原因**：

- 现有外部命令路径只适合输出诊断，不适合承载 hover/completion/definition 查询。
- 它默认读取磁盘文件，不能天然覆盖未保存 buffer。
- 这会把 LSP 正确性路径绑到另一个命令行入口上。

## 9. 推荐的最小实现顺序

### Phase A：文档先行

- 增加文档注释权威规范
- 修正文档中 `.ft` / LSP 的边界描述
- 增加 `feng lsp` 顶层命令规范

### Phase B：LSP 最小可用服务端

- 新增 `feng lsp` 路由
- 新增 LSP 服务端框架
- 复用现有前端 / semantic 做 diagnostics / hover / definition
- 支持“无本地 `.ft`”时直接源码分析

### Phase C：文档注释落地

- 词法 / 语法 / AST 承载文档注释
- 源码分析路径可返回文档注释
- hover 展示文档注释

### Phase D：`.ft` 文档注释与缓存消费补齐

- 补齐 `DOCS` section
- `.ft` 读取路径支持文档注释
- LSP 在缓存命中时返回与源码分析一致的文档信息

### Phase E：后续性能优化

- 是否需要为 workspace `.ft` 增加统一生产策略
- 本地缓存失效策略
- 增量分析与后台索引

## 10. 验收口径

首版 LSP 达标条件：

- 没有本地 workspace `.ft` 时，LSP 仍可工作。
- 有有效 workspace `.ft` 时，LSP 可直接消费。
- 文档注释在源码路径和缓存路径下都能得到一致结果。
- 未保存文件不要求先手动执行任何外部命令。
- 引入 LSP 后，禁止因为 LSP 需求改动 `codegen` 与 `runtime`，也不要求重构现有构建职责。

## 11. 一句话结论

Feng 首版 LSP 的正确做法不是“先把本地 `.ft` 生产链路做完，再接 LSP”，而是：

- **先让 LSP 直接复用现有源码分析能力跑通正确性**；
- **再把 workspace `.ft` 当作可选缓存逐步接入优化**；
- **禁止为了 LSP 触碰 `src/codegen/` 与 `src/runtime/`**；
- **如果必须改现有代码，就只改新增 `feng lsp`、内存文本覆盖前端、文档注释承载与 `.ft` 文档载荷这几处最小必要改动。**
